# Prompt

Modify the @specs/spark-ci-spec.md file as follows:

- use as prefix the format `<stage_id>--<pkg_id>--<job_id>`
  Because:
    - without it, reading the output file, there is no way to figure out which stage the job belongs to.
    - Without it there is no way to specify the same job ID in the same package file in different stages.

- Change the separator in the job prefixing from the dot `.` to double dashes `--` since the dot is not allowed by GitHub actions. 

- Forbid the use of `--` in job names (pkgs and project) and `id:` of pkg files (since it's our separator).

- The naming of `base.yaml` is confusing. Rename it everywhere to `configuration.yaml`.

- It should be possible to specify `pre-<stage>` and `post-<stage>` stages for each one of the stages defined in the configuration file.
  As an example say that the configuration file lists stages A, B and C. It should be possible to insert a job in a `pre-A` stage.

- The specifications mention that the package files will be present in a directory. This is too limiting for us. That's because we don't want to rely on the ordering of the operating system or alphabetical ordering. We need to explicitly list the order of the package files via repeated command line switches.

- It should be possible to specify the keyword needs in jobs in the project and pkg files. When present, these explicit dependencies are **preserved as is and merged** with the automatic stage-based dependencies.

# Spark CI Specification Changes Summary

## Version Update
- **From:** 2.0.0-draft (2026-02-10)
- **To:** 2.1.0-draft (2026-02-11)

## Configuration File Renaming
- `base.yml` renamed to `configuration.yml` throughout the specification
- CLI flag changed from `--base`/`-b` to `--conf`/`-c`

## Job ID Format Change
- **Old format:** `package.job-id` (e.g., `drupal.docker-php`)
- **New format:** `stage--package--job-id` (e.g., `build--drupal--docker-php`)
- **Separator changed:** from `.` to `--` (dots not allowed in GitHub Actions job IDs)
- Job IDs now include stage prefix for clearer identification and collision avoidance

## CLI Tool Changes
- **Tool name:** `sparkfabrik/spark-ci-loader` → `sparkfabrik/spark-ci`
- **Package loading:** Changed from automatic discovery of `pkg_*.yml` files to explicit `--pkg` flags (repeatable, order matters)
- **Removed:** `--pkg-dir` option
- **Added:** Multiple `--pkg` switches for explicit package ordering

## New Features

### Pre/Post Stage Hooks (Section 4.3.5)
- Added virtual stages `pre-<stage>` and `post-<stage>`
- Jobs can be inserted before or after any configured stage
- Examples: `pre-build`, `post-test`, `pre-deploy`
- Needs chains computed automatically for proper dependency ordering

### Explicit Dependencies Support (Section 5.3)
- Jobs can now declare explicit `needs` in source definitions
- Explicit dependencies are merged with automatic stage-based dependencies
- Job IDs in explicit needs must use prefixed names

### Enhanced Validation Rules
- Package IDs must not contain `--` (reserved separator)
- Job IDs must not contain `--` (reserved separator)
- Stage validation now includes pre/post stage base names
- Removed requirement for non-empty `steps` list

## Removed Sections
- **Section 9: Security Considerations** (secrets, action pinning, code injection)
- **Section 12: Rollout Plan** (phase 1-4 implementation timeline)
- JSON Schema component reference removed from architecture

## Simplified Sections
- **Lifecycle Management (6.2):** Now simply states generation happens via CLI similar to GitLab CI

## Assembly Process Updates
- Display names updated to reflect new `stage--package--job` format
