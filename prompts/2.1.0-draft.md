Modify the @specs/spark-ci-spec.md file as follows:

- use as prefix the format `<stage_id>--<pkg_id>--<job_id>`
  Because:
    - without it, reading the output file, there is no way to figure out which stage the job belongs to.
    - Without it there is no way to specify the same job ID in the same package file in different stages.

- Change the separator in the job prefixing from the dot `.` to double dashes `--` since the dot is not allowed by GitHub actions. 

- Forbid the use of `--` in job names (pkgs and project) and `id:` of pkg files (since it's our separator).

- It should be possible to specify `pre-<stage>` and `post-<stage>` stages for each one of the stages defined in the configuration file.

- The specifications mention that the package files will be present in a directory. This is too limiting for us. That's because we don't want to rely on the ordering of the operating system or alphabetical ordering. We need to explicitly list the order of the package files via repeated command line switches.

## Current State

Your `configuration.yml` defines three of the four GHA root-level keywords:

| Keyword | In `configuration.yml` | In `project.yml` |
|---------|----------------------|-------------------|
| `name` | Yes | No |
| `on` | Yes | No |
| `defaults` | Yes | No |
| `env` | Not mentioned | No |

The question is: **should `project.yml` be able to influence these root-level keywords, and if so, with what semantics?**

## Analysis of Each Keyword

Let me think through each one by its nature:

**`env`** (workflow-level) — This is a map. The merge semantics are unambiguous: deep merge, project keys win on conflict. This is exactly the same behavior as `env` at the job level, which your spec already handles. Adding this seems like a clear win with zero ambiguity.

**`on`** (triggers) — This is a map of maps/arrays. The tricky part: does a project want to *add* a trigger (e.g., add `workflow_dispatch` on top of the existing `push`/`pull_request`), or *replace* the entire trigger definition (e.g., "only trigger on tags, not on branches")? Both are realistic use cases.

**`defaults`** — Similar problem. A project might want to change `defaults.run.shell` from `bash` to `sh` (extend/override a scalar), or completely redefine what `defaults` means (replace).

**`name`** — Trivially a scalar. If the project sets it, it wins. Not much complexity here.

## The Core Tension

You already solved this exact problem at the job level with `extend` vs `replace`. The question is whether to reuse that same pattern at the root level.

Here are three approaches I see:

### Approach A: Simple top-level keys with implicit deep merge

```yaml
# project.yml
name: "My Project CI"       # scalar → replaces
env:                         # map → deep merges
  DEPLOY_ENV: staging
on:                          # map → deep merges
  workflow_dispatch: {}
defaults:                    # map → deep merges
  run:
    shell: sh
hooks:
  # ... existing syntax
```

The semantics follow the same deep merge rules you already defined in section 5.5: maps merge recursively (project wins on conflict), scalars replace. No new syntax needed.

**Pros:** Zero new concepts. Consistent with how `extend` already works at job level. Covers the "extend" use case naturally. For `env` this is exactly right. For `on`, adding `workflow_dispatch` to existing triggers is the merge case, which works.

**Cons:** No way to *replace entirely*. If a project wants to say "only trigger on tags, throw away the upstream `push`/`pull_request`", deep merge can't express that — it can only add keys or override existing ones. You can't *remove* a key with a merge.

### Approach B: Reuse extend/replace syntax for root-level keywords

Introduce a new top-level section in `project.yml` (say, `workflow:`) that mirrors the `extend`/`replace` semantics:

```yaml
# project.yml — extend case (deep merge)
workflow:
  env:
    DEPLOY_ENV: staging
  on:
    workflow_dispatch: {}

# project.yml — replace case (full substitution)
workflow:
  on:
    replace: true
    value:
      push:
        tags: ["v*"]
```

**Pros:** Explicitly handles both extend and replace. Clear intent.

**Cons:** The `replace: true` / `value:` syntax is a new pattern, different from how `extend`/`replace` work at job level. It feels bolted on. The job-level pattern uses `extend: { provided_by: ... }` which doesn't map cleanly to root-level keywords (there's no "provided_by" — the source is always `configuration.yml`).

### Approach C: Deep merge by default, with explicit replace via a sentinel

```yaml
# project.yml — deep merge (default for maps)
name: "My Project CI"
env:
  DEPLOY_ENV: staging
on:
  workflow_dispatch: {}

# project.yml — replace a specific root key
on:
  _replace: true      # sentinel key: discard upstream, use only what follows
  push:
    tags: ["v*"]
```

**Pros:** Compact. One small addition to the merge algorithm.

**Cons:** Magic sentinel keys are fragile. What if someone actually needs an `_replace` key in their `env` map? It pollutes the namespace. Also, this is a pattern that doesn't exist anywhere else in GitHub Actions or in your spec.

## My Recommendation

I'd lean toward **Approach A** (simple deep merge) with one practical observation:

**For `env` and `name`**: deep merge (or scalar replacement) is sufficient. There is no realistic scenario where a project needs to *remove* an env var set by `configuration.yml` — if they want a different value, they override it. If they don't want it, they just don't use it. Similarly, `name` is a scalar, so replacement is trivial.

**For `on` and `defaults`**: deep merge covers 90%+ of cases. The remaining case ("I want to completely replace the triggers") is genuinely rare, and when it happens, the project team can just set the `configuration.yml` triggers to the minimal common set and let the project file add what it needs. Or — and this is perhaps the cleanest escape hatch — document that for the rare "replace entirely" case, the project should use a different `configuration.yml` (since `--conf` is already a CLI flag).

So the concrete proposal would be:

```yaml
# project.yml
name: "My Project CI"          # optional, replaces configuration.yml name
env:                            # optional, deep merges with configuration.yml env
  DEPLOY_ENV: staging
on:                             # optional, deep merges with configuration.yml on
  workflow_dispatch: {}
defaults:                       # optional, deep merges with configuration.yml defaults
  run:
    shell: sh

hooks:
  # ... existing syntax unchanged
```

This means `configuration.yml` should also support `env` (it currently doesn't mention it). And `project.yml` gets four optional top-level keys alongside `hooks`.

The merge order is: `configuration.yml` → `project.yml` (project wins), using the exact same deep merge algorithm from section 5.5.

**What do you think? Does the "deep merge only, no replace for root-level keys" approach feel sufficient for your use cases, or do you have a concrete scenario where a project truly needs to wipe an entire root-level keyword?**

Let me think through this carefully.

## Moving the 4 keywords to packages/project

Your reasoning makes sense: `configuration.yml` becomes purely structural (stages, version, schema version), while packages and the project file own the "data" that flows into the generated workflow.

The assembly chain for root-level keywords would be:

```
pkg_1.yml → pkg_2.yml → ... → pkg_N.yml → project.yml
 (lowest)                                   (highest)
```

Using the same deep merge algorithm from section 5.5: maps merge recursively (right wins on conflict), scalars replace.

## Walking through each keyword

**`name`** — Scalar. Last one wins. If `pkg_base.yml` sets `name: Spark CI` and `project.yml` sets `name: My Project CI`, the output is `My Project CI`. Clean.

**`env`** — Map of scalars. Multiple packages contribute keys, project can override any of them. Deep merge. This is the one you expect to see in multiple packages. No issues.

**`defaults`** — Map of maps (e.g., `defaults.run.shell`, `defaults.run.working-directory`). Deep merge works. Realistically set once in a base package, project can override individual properties. Clean.

**`on`** — Map of maps/scalars. This is where I want to pause.

## One concern with `on`

Consider this scenario:

```yaml
# pkg_base.yml (the "base" package)
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
```

```yaml
# project.yml — project wants to also trigger on tags
on:
  push:
    tags: ["v*"]
```

After deep merge, `on.push` is itself a map, so it merges recursively:

```yaml
on:
  push:
    branches: [main, develop]   # from pkg_base
    tags: ["v*"]                # from project
  pull_request:
    branches: [main]            # from pkg_base
```

That works. But now consider:

```yaml
# project.yml — project wants different branches
on:
  push:
    branches: [main, release/*]
```

After deep merge:

```yaml
on:
  push:
    branches: [main, release/*]   # replaced — branches is a sequential array
  pull_request:
    branches: [main]              # preserved from pkg_base
```

This also works correctly because `branches` is a sequential array, so it gets fully replaced by the project's value. The project gets exactly what it asked for.

The one case that deep merge *cannot* express is: "remove `pull_request` entirely while keeping `push`." But given your constraint that `on` must be a map, there's no clean way to say "delete this key" in a merge-only world. I think this is acceptable — it's a rare edge case, and if a project truly doesn't want `pull_request` triggers, they can set it to an empty/restrictive value or use a different base package.

## On the map-only restriction for `on`

GitHub Actions allows these shorthand forms:

```yaml
on: push                    # scalar
on: [push, pull_request]    # list
```

Requiring only the map form:

```yaml
on:
  push: {}
  pull_request: {}
```

This is a good constraint. It means `on` is always a map, so merge behavior is always well-defined. You'd want to validate this early and fail with a clear error if someone uses the shorthand.

## One question about package ordering

Since packages are loaded in `--pkg` order and each one merges on top of the previous, the order matters for these root-level keywords. For `env`, if two packages define the same key with different values, the later package wins. This is deterministic and explicit (the user controls the `--pkg` order), but worth documenting.

For `name`, `defaults`, and `on` — which you say will realistically only appear in a base package — ordering is a non-issue in practice.

## Summary

I don't see a problem with this approach. The constraints are:

1. `configuration.yml` owns only `version` and `stages`
2. `name`, `defaults`, `on`, `env` live in packages and/or `project.yml`
3. `on` must be in map form (no scalar or list shorthand)
4. Assembly: packages merge left-to-right in `--pkg` order, then project merges on top
5. Merge uses the existing deep merge algorithm (maps merge recursively, arrays and scalars replace)

The semantics are consistent with what you already have at the job level, and the map-only restriction on `on` eliminates the only type ambiguity that could cause problems.
