Modify the @specs/spark-ci-spec-v2.0.0.md file as follows:

- Change the separator in the job prefixing from the dot `.` to double dashes `--` since the dot is not allowed by GitHub actions. 

- Also explicitly add a validation step that avoids `--` as project job name since it's our separator.

- Clarify that the following (i.e. defining the same job ID in different stages in a specific package file) should be illegal.
It was an ill-specified and corner case behavior that we want to avoid because:

  Rationale:

  - In the project file it would be not clear which one of the two jobs we want to extend, disable or replace.
  - Even without considering project files, it would still be unclear which job from the package is the one to consider, since the specs say that the final job id is the prefixed one `<package_name>.<job_id>` (An initial prototype developed with previous specifications ended up considering only the second job definition discarding the previous one.)

```yaml
# some pkg_something.yaml file.
hooks:
  stage_a:
    some_setup_job:
      # ...
    job_1:
      # ...
  stage_b:
    some_setup_job: # <---- validation error: same job id in 2 stages
      # ...
    job_2:
      # ...
```

- We should enable the three following root level GitHub Actions keywords, `env`, `on` and `permissions`, to be specified at the project level. Like the existing `defaults` keyword, they are passed through to GitHub Actions.

- The naming of `base.yaml` is confusing. Rename it everywhere to `configuration.yaml`.

- It should be possible from the project file to specify `pre-<stage>` and `post-<stage>` stages for each one of the stages defined in the configuration file.
  As an example say that the configuration file lists stages A, B and C. It should be possible from the project file to insert a job in a `pre-A` stage.

- The specifications mention that the package files will be present in a directory. This is too limiting for us. That's because we don't want to rely on the ordering of the operating system or alphabetical ordering. We need to explicitly list the order of the package files via repeated command line switches.
